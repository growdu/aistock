# AI量化交易系统资源要求文档

## 1. 文档目的

本文档基于 [product.md](/Users/growduduan/pythowork/aistock/docs/product.md)、[high_level_design.md](/Users/growduduan/pythowork/aistock/docs/high_level_design.md) 和 [detailed_design.md](/Users/growduduan/pythowork/aistock/docs/detailed_design.md)，重新规划面向`个人使用`的资源方案。

本版资源设计以低频、小规模、低成本为原则，适用于：

1. 每日交易次数不超过 `5 次`
2. 每次操作股票数量不超过 `3 支`
3. 以个人研究、回测、信号辅助和轻量自动交易为主
4. 模型采用 `4B 级小模型` 或更轻量模型

## 2. 设计原则

1. 优先单机部署，避免多机和分布式复杂度。
2. 优先 CPU 可运行，GPU 仅作为加速选项。
3. 优先轻量组件，能不用 Redis、Kafka、ELK 就不用。
4. 训练与交易解耦，盘中只做推理，不做大规模训练。
5. 优先本地可控和低月成本，而不是高可用架构。

## 3. 使用场景假设

本资源方案基于以下假设：

1. 股票池规模有限，通常是 `50-300 支` 重点观察标的，不是全市场高频扫描。
2. 盘中信号生成频率较低，以 `分钟级` 或更低频率为主。
3. 实盘只做个人账户，不涉及多账户、多用户、多租户。
4. 自动交易规模较小，主要是策略辅助决策或半自动执行。
5. 主要模型组合为：
   1. 传统量化模型：LightGBM、XGBoost、规则因子模型
   2. 小模型：4B 量级 LLM，用于公告/新闻摘要、情绪分析、解释增强

## 4. 总体资源规划

个人版建议采用下面的资源形态：

1. `1 台主机`
   用于开发、回测、数据采集、特征生成、在线推理、交易执行。
2. `1 份外部备份`
   用于数据库、模型文件、配置文件和交易日志备份。
3. `1 个实盘账户 + 1 个模拟账户`
   实盘与验证隔离。

不建议首期投入的资源：

1. 多台应用服务器
2. 独立数据库机
3. 独立训练机
4. Kafka、RabbitMQ 等消息队列
5. ELK 全套日志平台
6. Kubernetes 集群

## 5. 机器资源要求

## 5.1 推荐部署方案

### 方案 A：单台本地工作站

适合长期个人使用，成本最低。

推荐配置：

| 项目 | 推荐配置 |
| --- | --- |
| CPU | 8-12 核 |
| 内存 | 32 GB |
| 磁盘 | 1 TB SSD |
| GPU | 可选，8-16 GB 显存 |
| 系统 | Ubuntu 22.04 LTS / macOS / Windows + Linux 环境 |
| 用途 | 开发、回测、轻量训练、盘中推理、交易执行 |

说明：

1. 如果主要是传统量化模型加少量文本处理，`32 GB 内存 + SSD` 已足够起步。
2. 若 4B 小模型采用 `4bit` 或 `8bit` 量化，本地 `8-16 GB 显存` 通常可以覆盖推理。
3. 若没有 GPU，也可以只在 CPU 上运行，但推理速度会慢一些，适合低频使用。

### 方案 B：单台轻量云主机

适合希望机器长期在线、方便定时任务和盘中运行。

推荐配置：

| 项目 | 推荐配置 |
| --- | --- |
| CPU | 8 vCPU |
| 内存 | 16-32 GB |
| 磁盘 | 500 GB SSD |
| GPU | 默认不需要 |
| 带宽 | 10-20 Mbps |
| 系统 | Ubuntu 22.04 LTS |
| 用途 | 数据采集、回测、小模型推理、自动下单 |

说明：

1. 个人低频交易下，云主机不必上 GPU。
2. 若确实需要 4B 模型加速，可临时租用 GPU 云实例做离线处理，而不是长期持有。

## 5.2 最小可用配置

如果目标只是先跑通 MVP，最低建议：

| 项目 | 最低建议 |
| --- | --- |
| CPU | 4-8 核 |
| 内存 | 16 GB |
| 磁盘 | 300-500 GB SSD |
| GPU | 无 |
| 用途 | 数据采集、回测、规则策略、LightGBM、小规模推理 |

说明：

1. 这个配置可以先做回测、基础策略和轻量自动交易。
2. 若要本地稳定跑 4B 小模型推理，建议升级到 `32 GB 内存` 或增加 GPU。

## 5.3 推荐资源分配

单机内部建议按逻辑控制资源，而不是拆机器：

| 模块 | 建议资源占用 |
| --- | --- |
| 数据采集 | 1-2 CPU，4 GB 内存 |
| 回测任务 | 2-4 CPU，8-16 GB 内存 |
| 在线推理 | 2-4 CPU，4-8 GB 内存 |
| 数据库存储 | 2 CPU，4-8 GB 内存 |
| 交易执行 | 1 CPU，2 GB 内存 |

## 6. 模型资源要求

## 6.1 模型选择建议

个人版不建议把大模型作为核心交易预测引擎，建议角色分工如下：

1. `主预测模型`
   使用 LightGBM、XGBoost、时序特征模型，负责收益率、方向、排序信号。
2. `4B 小模型`
   用于公告摘要、新闻情绪、盘后复盘、信号解释，不直接决定下单。

原因：

1. 个人场景下交易频次低，4B 小模型做辅助足够。
2. 直接依赖 LLM 做交易信号，成本高且稳定性弱。
3. 传统模型更容易回测、调参和解释。

## 6.2 4B 小模型推理资源

如果采用量化后的 4B 模型，建议资源如下：

| 模式 | 建议资源 |
| --- | --- |
| CPU 推理 | 8 核 CPU + 32 GB 内存 |
| GPU 推理 | 8 GB 以上显存即可起步，12-16 GB 更稳妥 |
| 模型存储 | 5-15 GB，视量化方式而定 |

说明：

1. 4B 模型适合做文本理解和摘要，不适合在个人设备上高频并发服务化。
2. 盘中应避免反复对全量新闻做大规模推理，建议只对命中的少量股票文本增量处理。

## 6.3 训练建议

个人版不建议自训练 4B 模型，建议：

1. 直接使用现成开源小模型。
2. 如需适配金融语境，优先用 Prompt、模板和小规模 LoRA 微调。
3. 微调任务建议使用临时云 GPU，而不是本地常驻训练机。

## 7. 软件组件压缩方案

## 7.1 必需组件

| 组件 | 用途 | 是否必需 |
| --- | --- | --- |
| Python 3.10+ | 主开发语言 | 必需 |
| Pandas / Numpy | 数据处理 | 必需 |
| PostgreSQL 或 SQLite | 存储交易、配置、结果 | 必需 |
| LightGBM / XGBoost | 核心量化模型 | 必需 |
| Backtrader | 回测 | 必需 |
| Git | 版本管理 | 必需 |
| 定时任务工具 | 数据同步、盘前盘后任务 | 必需 |

说明：

1. 如果是纯个人单机，数据库可以先从 `SQLite` 起步。
2. 如果开始有较多历史数据、并发读写和交易记录，建议升级到 `PostgreSQL`。

## 7.2 建议组件

| 组件 | 用途 | 是否建议 |
| --- | --- | --- |
| PyTorch | 小模型推理或轻量微调 | 建议 |
| Docker | 环境一致性 | 建议 |
| Plotly / Streamlit | 本地可视化面板 | 建议 |
| Prometheus + Grafana | 监控 | 可选 |
| Redis | 缓存、幂等控制 | 可选 |

## 7.3 不建议首期引入的组件

1. Kafka
2. RabbitMQ
3. ELK 全套
4. Airflow 重型调度
5. Kubernetes
6. 独立模型管理平台

## 8. 账号资源要求

## 8.1 必需账号

| 账号类型 | 用途 | 说明 |
| --- | --- | --- |
| TuShare Pro 或等价数据源 | 日线、分钟线、财务、基础面 | 首期核心数据源 |
| 券商实盘账号 | 实际交易 | 必需 |
| 券商 API 权限 | 自动下单、查单、持仓查询 | 必需 |
| 模拟交易账号 | 联调和回测后的验证 | 必需 |
| Git 仓库账号 | 代码和文档管理 | 必需 |

说明：

1. 对个人产品来说，核心账号不是越多越好，而是要稳定。
2. 建议优先保证 `1 个稳定数据源 + 1 个稳定券商 API`。

## 8.2 建议账号

| 账号类型 | 用途 | 说明 |
| --- | --- | --- |
| 新闻/公告数据源账号 | 情绪分析 | 有预算再加 |
| 企业微信/钉钉机器人 | 告警推送 | 建议 |
| 云主机账号 | 长期开机部署 | 若本地机器不常在线 |
| 对象存储账号 | 备份模型和历史数据 | 建议 |

## 8.3 可省略账号

在个人场景中，以下资源可先不准备：

1. 多家行情源账号
2. 多个券商账号
3. 专门的日志平台账号
4. 制品仓库账号
5. 项目管理平台企业版账号

## 9. 存储资源要求

## 9.1 推荐容量

个人版首期建议：

| 数据类型 | 建议容量 |
| --- | --- |
| 历史行情数据 | 50-150 GB |
| 财务与主数据 | 10-20 GB |
| 新闻公告文本 | 20-100 GB |
| 特征与回测结果 | 50-200 GB |
| 模型文件 | 20-50 GB |
| 交易与日志数据 | 10-30 GB |
| 备份空间 | 200-500 GB |

说明：

1. 个人交易频次很低，真正增长快的是历史行情和特征快照，不是订单数据。
2. 订单和持仓数据量非常小，不需要大规模数据库磁盘。

## 9.2 备份建议

最少保留以下备份：

1. 数据库每日备份 `1 份`
2. 模型与配置文件每次变更后备份 `1 份`
3. 交易日志和信号日志按日归档
4. 重要数据同步到移动硬盘或对象存储

## 10. 网络资源要求

个人版网络要求很低，重点是稳定，而不是极致低延迟。

建议如下：

1. 稳定宽带即可，建议 `20 Mbps` 以上。
2. 若使用券商白名单 API，需准备固定公网 IP 或固定出口方案。
3. 若本地机器经常休眠或断网，建议改为云主机运行盘中任务。

## 11. 安全资源要求

个人版也必须保留最基本的安全措施：

1. 券商账号、API Key、数据库密码必须脱离代码明文存储。
2. 至少使用 `.env + 本地加密文件` 或系统密码库管理密钥。
3. 关键交易日志必须保留，便于复盘。
4. 下单前保留人工确认开关，避免策略异常直接连发。
5. 建议设置每日最大亏损阈值、单股仓位阈值和单日最大交易次数阈值。

## 12. 最小落地资源清单

如果目标是尽快上线一个个人可用版本，最小资源清单建议如下：

1. `1 台主机`
   配置建议：`8 核 CPU / 32 GB 内存 / 1 TB SSD`
2. `1 个核心数据源账号`
   例如 TuShare Pro 或同等级服务
3. `1 个券商实盘账号`
4. `1 个券商模拟账号`
5. `1 套 Git 仓库`
6. `1 套本地或轻量数据库`
   优先 `SQLite`，后续可升级 `PostgreSQL`
7. `1 个 4B 小模型运行环境`
   本地量化推理或云端按次使用
8. `1 份外部备份空间`
   移动硬盘、NAS 或对象存储均可

## 13. 推荐落地版本

如果希望兼顾成本和稳定性，推荐资源组合如下：

1. 主机：`8-12 核 CPU / 32 GB 内存 / 1 TB SSD`
2. 数据库：`PostgreSQL`
3. 模型：`LightGBM + 4B 小模型`
4. 展示层：`Streamlit` 或简单 Web 面板
5. 定时任务：`cron` 或轻量调度脚本
6. 告警：`企业微信/钉钉机器人`
7. 备份：`本地 + 云端双份`

这是当前约束下最均衡的方案：

1. 足够支撑低频个人交易
2. 维护成本低
3. 不会因为系统过重拖累策略迭代
4. 后续还能逐步扩展到更完整的自动化体系

## 14. 后续扩容触发条件

只有在出现以下情况时，才建议扩容：

1. 每日跟踪股票池扩大到 `500 支以上`
2. 盘中推理频率提高到 `秒级`
3. 开始接入多个账户
4. 开始长期微调或训练更大的模型
5. 单机回测耗时和存储开始明显成为瓶颈

在此之前，不建议过早采购额外机器或重型平台。

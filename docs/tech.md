# AI量化交易系统技术架构选型文档

## 1. 文档目的

本文档基于 [product.md](./product.md)、[high_level_design.md](./high_level_design.md)、[detailed_design.md](./detailed_design.md) 和 [resource.md](./resource.md)，给出当前需求下的技术架构选型结论。

本文档的目标不是列出所有可选技术，而是结合当前约束，明确“现在应该用什么”。

## 2. 当前约束

当前技术选型必须服从以下约束：

1. 产品面向`个人用户`，不是团队协作平台。
2. 每日交易次数不超过 `5 次`。
3. 每次操作股票不超过 `3 支`。
4. 系统以`低频、轻量、稳定、低成本`为目标。
5. 优先`单机部署`，避免分布式系统复杂度。
6. 模型侧允许使用 `4B 小模型`，但不适合作为主交易引擎。

## 3. 选型原则

1. 用最少组件跑通“数据 -> 因子 -> 预测 -> 风控 -> 交易 -> 复盘”闭环。
2. 优先选择 Python 生态，减少跨语言和跨进程成本。
3. 优先选择成熟、可验证、易回测的量化工具，而不是新潮但重型的基础设施。
4. 线上只保留必要的推理和交易逻辑，训练尽量离线化。
5. 对个人场景来说，可维护性优先于理论扩展性。

## 4. 总体架构选型

## 4.1 架构风格

选型结论：采用`单体式模块化架构`，而不是微服务架构。

推荐结构：

1. `data`：数据采集、清洗、主数据管理
2. `feature`：因子计算、特征构建
3. `model`：训练、模型加载、推理
4. `strategy`：信号生成、仓位计算
5. `risk`：风控检查、阈值控制
6. `broker`：券商适配、订单执行
7. `backtest`：历史回测、参数验证
8. `report`：复盘、可视化、日报
9. `app`：CLI、调度入口、配置管理

选择原因：

1. 当前并发量极低，微服务拆分没有收益，只有运维负担。
2. 单体架构更适合个人长期维护。
3. 单机内模块化足以支持后续局部重构。

不选微服务的原因：

1. 需要额外引入服务发现、消息队列、监控链路和部署编排。
2. 当前交易规模和用户规模不足以支撑这类复杂度。

## 4.2 部署方式

选型结论：采用`单机部署 + 定时任务 + 本地数据库`。

推荐形态：

1. 本地工作站长期运行，或
2. 单台轻量云主机长期在线运行

部署建议：

1. 使用 `Python 虚拟环境` 或 `Docker Compose` 管理运行环境。
2. 使用 `cron` 或轻量调度脚本执行盘前、盘中、盘后任务。
3. 所有核心模块运行在同一台机器上。

不选 Kubernetes、Nomad、多机编排的原因：

1. 完全超出当前规模需要。
2. 会显著增加部署、监控和恢复复杂度。

## 5. 编程语言与工程框架选型

## 5.1 主语言

选型结论：`Python 3.10+`

选择原因：

1. 数据处理、回测、机器学习、量化工具链最成熟。
2. 与 Pandas、NumPy、LightGBM、PyTorch、Backtrader 无缝集成。
3. 开发成本最低，适合个人快速试错。

## 5.2 服务接口形态

选型结论：首期以`CLI + 本地任务入口`为主，必要时补一个轻量 Web API。

推荐方式：

1. `Typer` 或 `argparse`：实现命令行入口
2. `FastAPI`：仅在需要 Web 查询、看板、控制台接口时引入

原因：

1. 个人使用不需要先上完整前后端分离系统。
2. 绝大多数任务是批处理和定时任务，不是高并发 API。

## 5.3 配置管理

选型结论：`Pydantic Settings + .env + YAML`

用途划分：

1. `.env`：账号、密钥、数据库连接
2. `YAML`：策略参数、风控阈值、数据源配置
3. `Pydantic Settings`：统一装载和校验配置

## 6. 数据层选型

## 6.1 事务数据库

选型结论：`PostgreSQL` 作为主数据库，`SQLite` 可用于最初原型阶段。

推荐策略：

1. 原型阶段：可以先用 `SQLite`
2. 准备长期使用或接入实盘后：升级为 `PostgreSQL`

选择 PostgreSQL 的原因：

1. 结构化查询能力强，适合订单、持仓、信号、风控和配置数据。
2. 稳定、成熟、成本低。
3. 比 SQLite 更适合长期积累交易与回测数据。

不选 MongoDB、ClickHouse 的原因：

1. 当前数据量不大，不需要专门的列式分析库或文档库。
2. 会引入额外运维成本。

## 6.2 文件存储

选型结论：`Parquet + 本地目录分层存储`

建议目录：

1. `data/raw`
2. `data/clean`
3. `data/features`
4. `data/models`
5. `data/reports`
6. `data/backups`

选择原因：

1. Parquet 对历史行情、特征快照和回测结果更省空间。
2. 便于用 Pandas 直接读写。
3. 对个人项目足够简单透明。

## 6.3 缓存层

选型结论：首期`不强制引入 Redis`。

替代方案：

1. 本地内存缓存
2. PostgreSQL 表状态
3. 文件级缓存

何时再引入 Redis：

1. 盘中任务明显增多
2. 幂等控制和实时状态管理开始复杂
3. 需要跨进程共享短期状态

## 7. 数据采集选型

## 7.1 行情与基础面数据源

选型结论：首期优先 `TuShare Pro`。

原因：

1. 覆盖日线、分钟线、财务、基础面数据。
2. Python 接入成本低。
3. 对个人开发足够友好。

## 7.2 新闻与公告数据

选型结论：首期采用`轻量接入`，不强依赖重型商业舆情系统。

推荐方式：

1. 上市公司公告源
2. 少量新闻源
3. 对进入观察池的标的做增量文本处理

原因：

1. 个人场景没必要实时跑全市场舆情分析。
2. 4B 小模型对少量文本做解释和摘要更合适。

## 8. 特征与量化研究选型

## 8.1 数据处理框架

选型结论：`Pandas + NumPy`

原因：

1. 开发效率高。
2. 数据规模在个人场景内完全可控。
3. 与 Backtrader、LightGBM、PyTorch 都容易衔接。

何时考虑 Polars：

1. 当分钟级历史数据量增大，Pandas 明显成为瓶颈时。

## 8.2 因子计算方式

选型结论：以`自定义因子函数 + 批量计算脚本`为主。

推荐实践：

1. 技术因子：动量、波动率、均线偏离、量比、换手率
2. 基本面因子：成长性、估值、研发投入、盈利质量
3. 市场因子：板块强弱、指数相对表现、资金流
4. 文本因子：公告情绪、新闻摘要标签

原因：

1. 当前场景更需要可解释性和可控性。
2. 先做少量高质量因子，比堆大量因子更有效。

## 9. 模型层选型

## 9.1 主预测模型

选型结论：`LightGBM` 作为首选主模型，`XGBoost` 作为备选对照模型。

原因：

1. 对结构化因子数据非常合适。
2. 训练快，调参成本低。
3. 对个人项目更容易形成稳定回测闭环。
4. 比直接上 Transformer/LSTM 更适合当前资源约束。

使用场景：

1. 涨跌方向预测
2. 收益率排序
3. 资金流强弱评分

## 9.2 时序模型

选型结论：首期`不把 LSTM/Transformer 作为主线`，仅作为后续增强选项。

原因：

1. 训练资源更高。
2. 个人数据治理和样本质量不足时，复杂模型未必更优。
3. 当前目标是先做稳定系统，不是追求最复杂模型。

## 9.3 小模型选型

选型结论：采用`4B 级量化小模型`作为文本增强模块，而不是核心预测模块。

推荐用途：

1. 公告摘要
2. 新闻情绪分析
3. 异常事件解释
4. 交易信号的人类可读说明

推荐运行方式：

1. 本地量化推理
2. 离线批处理
3. 只处理命中的少量股票文本

不建议的用法：

1. 直接让 LLM 决定买卖
2. 对全市场新闻做高频全量推理

## 9.4 深度学习框架

选型结论：`PyTorch`

原因：

1. 生态成熟。
2. 小模型推理、LoRA 微调和文本任务更方便。
3. 与 HuggingFace 生态兼容性好。

不选 TensorFlow 作为首选的原因：

1. 当前没有明显收益。
2. 在个人研究和小模型实践中，PyTorch 更灵活。

## 10. 回测与研究框架选型

## 10.1 回测框架

选型结论：`Backtrader`

原因：

1. Python 集成自然。
2. 社区成熟，适合个人项目。
3. 容易加入滑点、手续费、仓位和风控规则。

不选 JoinQuant/RQAlpha 作为主依赖的原因：

1. 平台依赖更重。
2. 项目希望本地可控，减少外部平台绑定。

## 10.2 参数优化

选型结论：首期使用`网格搜索 + 人工筛选`，暂不引入复杂 AutoML 平台。

原因：

1. 当前策略数量少。
2. 手工验证更容易避免过拟合。

## 11. 交易执行层选型

## 11.1 券商接入

选型结论：采用`自定义 Broker Adapter` 统一封装券商 API。

推荐接口：

1. `place_order`
2. `cancel_order`
3. `query_order`
4. `query_positions`
5. `query_balance`
6. `query_fills`

原因：

1. 避免策略层直接依赖某一家券商的字段和异常语义。
2. 后续更换券商时改动范围最小。

## 11.2 执行模式

选型结论：`目标仓位驱动 + 限价单优先 + 人工确认可开关`

原因：

1. 当前交易频次低，不需要复杂的执行算法。
2. 个人实盘最怕误下单，人工确认开关必须保留。
3. 目标仓位模式更容易做风控和复盘。

## 12. 风控架构选型

选型结论：采用`规则型风控引擎`，不单独做机器学习风控。

首期规则：

1. 单股最大仓位
2. 单次买入最大金额
3. 单日最大交易次数
4. 单日最大亏损阈值
5. 低流动性禁买
6. 重大负面公告禁买

原因：

1. 规则型风控更透明。
2. 个人场景最重要的是可理解、可验证、可手动覆盖。

## 13. 展示与交互层选型

## 13.1 本地可视化

选型结论：`Streamlit` 作为首选轻量界面方案。

用途：

1. 观察股票池
2. 查看信号结果
3. 查看持仓与收益
4. 查看公告摘要和模型解释
5. 查看回测结果

原因：

1. 开发快。
2. 单人使用足够。
3. 比自建前后端分离系统简单很多。

## 13.2 图表组件

选型结论：`Plotly`

原因：

1. 交互图能力够用。
2. 与 Streamlit 集成方便。

## 14. 调度与任务编排选型

选型结论：`cron + Python 脚本任务`

推荐任务：

1. 盘前数据更新
2. 开盘后定时信号计算
3. 盘后回写订单与持仓
4. 每日备份
5. 每周回测和模型重训

不选 Airflow 的原因：

1. 当前任务数量少。
2. 维护成本远高于收益。

## 15. 日志、监控与告警选型

## 15.1 日志

选型结论：`Python logging + 本地文件滚动`

日志分类：

1. 业务日志
2. 订单日志
3. 风控日志
4. 异常日志

## 15.2 监控

选型结论：首期以`轻量健康检查 + 关键指标记录`为主。

推荐做法：

1. 记录每日任务成功/失败
2. 记录数据更新时间
3. 记录订单执行结果
4. 记录组合净值和回撤

可选增强：

1. 后续再加 `Prometheus + Grafana`

## 15.3 告警

选型结论：`企业微信 / 钉钉机器人`

告警场景：

1. 下单失败
2. 风控阻断
3. 数据采集失败
4. 日亏损超阈值

## 16. 安全选型

选型结论：采用`轻量但严格`的本地安全方案。

推荐方案：

1. 密钥放在 `.env` 或系统密钥库，不写死在代码中。
2. 本地数据库和备份做最小权限控制。
3. 交易接口默认关闭，只有运行时明确开启。
4. 所有真实下单行为保留审计日志。

## 17. 最终技术栈结论

在当前需求下，推荐技术栈如下：

| 层级 | 选型 |
| --- | --- |
| 主语言 | Python 3.10+ |
| 工程入口 | Typer / CLI |
| 可选 API | FastAPI |
| 数据处理 | Pandas + NumPy |
| 主数据库 | PostgreSQL |
| 原型数据库 | SQLite |
| 文件存储 | Parquet |
| 核心模型 | LightGBM |
| 备选模型 | XGBoost |
| 小模型 | 4B 量化模型 |
| 深度学习框架 | PyTorch |
| 回测框架 | Backtrader |
| 展示层 | Streamlit + Plotly |
| 调度 | cron |
| 日志 | Python logging |
| 告警 | 企业微信 / 钉钉机器人 |
| 部署方式 | 单机部署 |

## 18. 当前阶段不建议采用的技术

以下技术在当前阶段不建议引入：

1. 微服务架构
2. Kubernetes
3. Kafka / RabbitMQ
4. ELK 全套
5. Airflow
6. 大模型作为主交易引擎
7. 自训练大型 Transformer 时序模型

## 19. 实施优先级建议

建议按以下顺序落地：

1. `Python + PostgreSQL/SQLite + Parquet` 跑通数据存储
2. `Pandas + LightGBM + Backtrader` 跑通研究和回测
3. `Broker Adapter + 风控规则` 跑通模拟交易
4. `4B 小模型 + Streamlit` 增强文本理解和可视化
5. 最后再补监控、告警和实盘自动化

这个顺序符合当前需求的核心逻辑：

1. 先把交易闭环做稳
2. 再把解释能力做强
3. 最后再考虑工程增强

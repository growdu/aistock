# AI量化交易系统实现计划

## 1. 文档目的

本文档用于将当前个人版 AI 量化交易系统从“设计与骨架阶段”推进到“可运行、可验证、可逐步接实盘”的实施阶段。

本文档重点回答四个问题：

1. 先做什么
2. 每阶段交付什么
3. 怎么判断阶段完成
4. 哪些风险需要提前控制

## 2. 当前基线

当前仓库已经具备：

1. 产品需求文档
2. 概要设计与详细设计
3. 资源规划和技术选型
4. 单体式 Python 工程骨架
5. CLI 入口
6. **P1：真实 Tushare 数据接入（12 个接口、增量 UPSERT 同步）**
7. **P2：完整特征工程（81 因子 + 9 标签、训练/推理口径一致）**
8. **P3：基线模型训练（LightGBM/XGBoost、时间切分、IC+AUC）**
9. **P4：策略、风控与回测闭环（多因子排序、仓位分配、完整成本模型）**
10. **P5：券商适配（SimBroker 模拟交易、QMTBroker 实盘、ExecutionEngine）**
11. **P6：可视化（Streamlit 看板：权益曲线/持仓/交易日志/风控）**

当前仓库尚未完成（下一步方向）：

1. 真实数据验证（运行 sync-data 并验证数据质量）
2. 回测指标验证（IC、胜率、夏普合理性检验）
3. QMT 实盘对接（在 Windows 上配置）
4. 生产级盯盘和告警

## 3. 实施原则

1. 先打通研究闭环，再接交易闭环。
2. 先用真实数据替换占位逻辑，再提升模型复杂度。
3. 先做可验证的简单版本，不直接追求“最强模型”。
4. 所有阶段都必须有可执行验收标准。
5. 任何实盘能力必须在模拟验证后再启用。

## 4. 总体阶段划分

建议分为 7 个阶段推进：

1. `P0`：工程基础完善
2. `P1`：真实数据接入与存储
3. `P2`：特征工程与数据集构建
4. `P3`：基线模型训练与推理
5. `P4`：策略、风控与回测闭环
6. `P5`：模拟交易与券商适配
7. `P6`：可视化、告警与运维增强

## 5. 分阶段实施计划

## 5.1 P0 工程基础完善

### 目标

把当前骨架整理成一个可持续开发的基础工程。

### 核心任务

1. 完善配置体系，明确 `.env`、`settings.yaml`、默认值和优先级。
2. 完善数据库基础设施，补充建表、初始化和迁移机制。
3. 增加基础日志能力和统一错误处理。
4. 增加基础测试框架和最小冒烟测试。
5. 增加本地开发脚本和环境检查脚本。

### 交付物

1. 可稳定安装的 Python 工程
2. 数据库初始化命令可重复执行
3. 最小测试集
4. 基础开发脚本

### 验收标准

1. `pip install -e .` 成功
2. `aistock init-db` 可重复执行且不报错
3. 基础 CLI 命令能跑通
4. 至少有一组自动化冒烟测试

### 依赖

1. 当前已存在工程骨架

### 风险

1. 如果配置体系不先稳定，后续数据和模型阶段会反复返工

## 5.2 P1 真实数据接入与存储

### 目标

用真实数据替换当前占位数据同步逻辑。

### 核心任务

1. 接入 TuShare 客户端和认证流程。
2. 实现交易日历、股票主数据、日线行情、分钟线行情、基础财务数据同步。
3. 建立标准化数据落地格式。
4. 建立原始数据和清洗后数据的本地目录规范。
5. 补充数据库表结构：
   1. `security_master`
   2. `market_bar_1d`
   3. `market_bar_1m`
   4. `financial_indicator`
   5. 必要的任务状态表

### 交付物

1. 真实 TuShare 数据同步命令
2. 可查询的主数据和行情数据
3. 数据落地目录与表结构

### 验收标准

1. 能同步指定股票池最近一段时间的日线和分钟线
2. 能同步基础财务指标
3. 数据可重复更新，不产生明显重复脏数据
4. 本地 Parquet 与数据库内容一致

### 依赖

1. P0 完成
2. 有效的 TuShare Token

### 风险

1. 数据源限流
2. 数据字段口径不一致
3. 分钟线数据量增长较快

## 5.3 P2 特征工程与数据集构建

### 目标

建立正式因子工程和训练/推理共享特征层。

### 核心任务

1. 实现技术因子：
   1. 动量
   2. 波动率
   3. 均线偏离
   4. 成交量和换手率相关指标
2. 实现基本面因子：
   1. 营收增速
   2. 净利润增速
   3. 研发投入占比
   4. 估值分位
3. 实现市场因子：
   1. 板块强弱
   2. 指数相对表现
   3. 资金流代理指标
4. 设计标签生成逻辑：
   1. 未来 1 日收益
   2. 未来 N 日方向
5. 建立训练样本导出能力。
6. 建立推理特征快照能力。

### 交付物

1. 正式特征构建模块
2. 训练样本表或 Parquet 文件
3. 推理特征快照输出

### 验收标准

1. 能为指定股票池批量生成特征
2. 训练样本中无明显未来函数泄露
3. 同一只股票的历史特征与推理特征口径一致

### 依赖

1. P1 完成

### 风险

1. 因子数量过多导致维护成本上升
2. 特征定义不稳定导致回测和实盘不一致

## 5.4 P3 基线模型训练与推理

### 目标

建立第一个可回测、可解释、可重复训练的基线模型。

### 核心任务

1. 使用 LightGBM 建立基线收益预测或排序模型。
2. 建立训练、验证、测试时间切分逻辑。
3. 输出训练指标和样本外验证指标。
4. 保存模型文件和特征列清单。
5. 实现批量推理入口。
6. 建立模型版本管理规则。
7. 增加 XGBoost 对照实验。

### 交付物

1. 基线训练脚本
2. 模型文件
3. 推理命令
4. 训练报告

### 验收标准

1. 能完成一轮全流程训练
2. 能对指定观察池生成预测分数
3. 模型版本、特征版本和数据版本可追溯
4. 训练和推理结果可落盘

### 依赖

1. P2 完成

### 风险

1. 样本规模不足
2. 回测表现和验证表现偏差较大
3. 过拟合

## 5.5 P4 策略、风控与回测闭环

### 目标

把模型预测转为可回测的策略决策，并建立完整研究闭环。

### 核心任务

1. 定义候选池过滤逻辑。
2. 定义排序和选股逻辑。
3. 定义目标仓位逻辑。
4. 定义规则型风控：
   1. 单股仓位限制
   2. 单日交易次数限制
   3. 单日最大亏损限制
   4. 低流动性限制
5. 补充回测引擎：
   1. 手续费
   2. 滑点
   3. 仓位调整
   4. 简化成交限制
6. 输出回测指标和报告。

### 交付物

1. 正式策略引擎
2. 风控规则集
3. 回测报告输出

### 验收标准

1. 能对历史数据完成完整回测
2. 能输出收益、回撤、胜率、换手率等指标
3. 风控规则在回测中生效
4. 回测结果可重复

### 依赖

1. P3 完成

### 风险

1. 回测过于理想化
2. 交易成本低估
3. 风控规则过松或过严
